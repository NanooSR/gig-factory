# Gig Factory Hermes Rules

## Mandatory startup order
This repo uses a SQLite-first continuity layer. On every fresh Hermes session in this repo:
1. Query `C:/Users/Ryan/gig-factory/.hermes/continuity/gig_factory_continuity.sqlite3` first.
2. Read tables in this exact order:
   1. `current_handoff_summary`
   2. `resume_queue`
   3. `active_workflows`
   4. `automation_state`
   5. `query_playbook`
   6. `database_directory`
   7. `tasks`
   8. `blockers`
   9. latest handoff doc
   10. only then deeper files/code as needed
3. Prefer the helper scripts before broad file reads:
   - `python scripts/continuity/resume_summary.py`
   - `python scripts/continuity/db_health_check.py`
   - `python scripts/continuity/workflow_status.py`
4. Do **not** start with giant file reads or whole-repo scans unless the DB/handoff says they are needed.

## Browser/auth workflow
- For GitHub, Vercel, or other authenticated browser work, use the dedicated visible Chrome CDP flow.
- Hermes config target: `browser.cdp_url = http://127.0.0.1:9222`
- Launch browser via: `scripts/browser/launch-hermes-chrome.cmd`
- Dedicated Chrome user-data-dir: `C:/Users/Ryan/AppData/Local/hermes/browser-profiles/cdp-chrome`
- Verify CDP with `http://127.0.0.1:9222/json/version` before browser automation.
- The user must sign in inside the Hermes-controlled Chrome window, not their everyday Chrome profile.
- If browser tools/config changes were made after the current session started, use `/reset` or start a fresh Hermes session before expecting browser tools to attach.

## Safety
- Do not touch unrelated GitHub repos/projects.
- Do not store passwords, tokens, cookies, or secrets in the SQLite DB or docs.
- Keep continuity state current after material changes.
